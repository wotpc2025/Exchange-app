# คู่มือแยกตามโมดูล (พร้อม Data Flow)

เอกสารนี้สรุปแยกเป็นโมดูลเพื่อให้รู้ว่า request ไหลอย่างไรจากหน้าเว็บไป DB

## โมดูล 1: สินค้า (Items)

ไฟล์หลัก:
- หน้า: [app/page.js](app/page.js), [app/items/[id]/page.js](app/items/[id]/page.js), [app/add-item/page.js](app/add-item/page.js), [app/my-items/page.js](app/my-items/page.js)
- API: [app/api/items/route.js](app/api/items/route.js), [app/api/items/[id]/route.js](app/api/items/[id]/route.js)

Flow:
1. หน้าแรกเรียก GET /api/items
2. API query ตาราง items (และ fallback สำหรับ schema เก่า)
3. หน้าเว็บกรอง approval_status + status ก่อนแสดง
4. ตอนเพิ่มประกาศ หน้า add-item เรียก POST /api/items
5. API ตรวจ session และบันทึก items + item_images

จุดที่ควรระวัง:
- approval_status กับ status มีผลต่อการมองเห็นบนหน้าแรก
- ถ้าเพิ่ม field ใหม่ ต้องแก้ทั้ง SELECT/INSERT และหน้าแสดงผล
- รูปหลายรูปต้องเช็กทั้ง table item_images และ fallback image_url

## โมดูล 2: คำขอแลก (Requests)

ไฟล์หลัก:
- หน้า: [app/requests/page.js](app/requests/page.js), [app/items/[id]/page.js](app/items/[id]/page.js)
- API: [app/api/requests/route.js](app/api/requests/route.js), [app/api/requests/[id]/route.js](app/api/requests/[id]/route.js), [app/api/requests/[id]/confirm/route.js](app/api/requests/[id]/confirm/route.js)

Flow:
1. ผู้ใช้กดทักจากหน้าสินค้า
2. หน้าเรียก POST /api/requests เพื่อสร้างหรือ reuse request เดิม
3. หน้า requests เรียก GET /api/requests?user=email เพื่อโหลดรายการ
4. เจ้าของโพสต์กดรับ/ปฏิเสธผ่าน PUT /api/requests/[id]
5. ทั้งสองฝ่ายกดยืนยันสำเร็จผ่าน POST /api/requests/[id]/confirm
6. เมื่อยืนยันครบ 2 ฝ่าย ระบบปิดดีล และอัปเดต items.status = exchanged

จุดที่ควรระวัง:
- endpoint สร้าง request ตอนนี้รับ requester_email/owner_email จาก client
- ถ้าจะ harden ความปลอดภัย ควรย้ายไปอ้าง session ฝั่ง server เป็นหลัก
- การเปลี่ยนสถานะมีผลต่อโมดูล chat และ meeting

## โมดูล 3: แชทแลกของ (Exchange Chat)

ไฟล์หลัก:
- หน้า: [app/chat/[requestId]/page.js]
- API: [app/api/chat/[requestId]/route.js](app/api/chat/[requestId]/route.js)

Flow:
1. หน้าแชทเรียก GET /api/chat/[requestId] เพื่อโหลด request + messages
2. API ตรวจสิทธิ์ owner/requester/admin
3. หน้าโพสต์ข้อความด้วย POST /api/chat/[requestId]
4. API sanitize ข้อความ + rate limit + บันทึก messages
5. หน้า polling ทุก 3 วินาทีเพื่อ refresh

จุดที่ควรระวัง:
- แชทรูปต้องรองรับ messages.image_url ขนาดใหญ่
- ถ้าเจอ Unknown column ให้เช็ก migration
- polling ถี่เกินไปจะเพิ่มโหลด DB

## โมดูล 4: นัดหมาย (Meeting)

ไฟล์หลัก:
- หน้า: [app/chat/[requestId]/page.js]
- API: [app/api/requests/[id]/meeting/route.js](app/api/requests/[id]/meeting/route.js)
- SQL: [db/phase3_migrations.sql](db/phase3_migrations.sql)

Flow:
1. หน้าแชทโหลดนัดด้วย GET /api/requests/[id]/meeting
2. API ตรวจว่า user เป็น owner/requester ของ request
3. POST สำหรับเสนอนัดใหม่ (location + meetTime)
4. PUT สำหรับ confirm/cancel/done
5. ทุก action ส่ง notification ให้คู่สนทนา

จุดที่ควรระวัง:
- ต้องมีตาราง exchange_meetings และคอลัมน์ครบ
- ปัญหา collation ในการ JOIN email อาจทำให้ขึ้น 500
- ต้องทดสอบ timezone ตอน parse เวลานัด

## โมดูล 5: แจ้งเตือน (Notifications)

ไฟล์หลัก:
- Component: [components/NotificationBell.js](components/NotificationBell.js)
- หน้า: [app/notifications/NotificationsClient.js](app/notifications/NotificationsClient.js)
- API: app/api/notifications/*

Flow:
1. หน้าแรก mount NotificationBell
2. component เรียก GET /api/notifications และ polling ทุก 15 วินาที
3. ผู้ใช้กดอ่านทีละรายการหรืออ่านทั้งหมด
4. หน้ารวม notifications แสดงรายการพร้อมตัวกรอง/แบ่งหน้า

จุดที่ควรระวัง:
- policy ปัจจุบัน: กระดิ่งแสดงเฉพาะ navbar หน้าแรก
- ถ้าจะเปิดใช้หลายหน้า ให้คุยเรื่อง UX ก่อน เพราะมี polling

## โมดูล 6: ซัพพอร์ต (Support)

ไฟล์หลัก:
- หน้า: [app/support/page.js](app/support/page.js), [app/support/[id]/page.js](app/support/[id]/page.js)
- API: [app/api/support/conversations/route.js](app/api/support/conversations/route.js), [app/api/support/conversations/[id]/route.js](app/api/support/conversations/[id]/route.js), [app/api/support/conversations/[id]/close/route.js](app/api/support/conversations/[id]/close/route.js)

Flow:
1. นักศึกษาเปิดเคสใหม่จากหน้า support
2. POST /api/support/conversations สร้างห้อง + ข้อความแรก
3. เข้า room แล้ว GET /api/support/conversations/[id]
4. ส่งข้อความผ่าน POST /api/support/conversations/[id]
5. แอดมินตอบครั้งแรก ระบบ assign admin_email
6. นักศึกษากดจบเคสผ่าน POST /api/support/conversations/[id]/close
7. ระบบอัปเดตสถานะเคส และสถิติแอดมิน

จุดที่ควรระวัง:
- กติกาปัจจุบันคือ student เท่านั้นที่ปิดเคส
- ถ้า DB schema เก่า endpoint close มี fallback บางส่วน
- ควรเช็กฟิลด์สถิติ admin ก่อน deploy

## โมดูล 7: แอดมิน (Admin)

ไฟล์หลัก:
- หน้า: app/admin/*
- API ตัวอย่าง: [app/api/admin/items/route.js](app/api/admin/items/route.js), [app/api/admin/reports/route.js](app/api/admin/reports/route.js), [app/api/admin/users/route.js](app/api/admin/users/route.js)

Flow รวม:
1. หน้าแอดมินเรียก API ที่ requireAdmin
2. API query ตารางหลัก + join/subquery ตามสถิติ
3. หน้าแสดง list เพื่อ moderate (approve/reject/ban/warn/close)
4. action สำคัญบางส่วนบันทึก audit logs

จุดที่ควรระวัง:
- query หนัก มี subquery เยอะ ต้องพึ่ง index
- ทุก route ควรเช็ก guard admin ก่อน query
- ถ้าเพิ่ม action admin ใหม่ ควรเพิ่ม log ใน admin_audit_logs

## โมดูล 9: Auth/Guard/Security

ไฟล์หลัก:
- [lib/auth.js](lib/auth.js)
- [lib/api-guards.js](lib/api-guards.js)
- [lib/security.js](lib/security.js)

Flow:
1. Route เรียก requireSessionOrThrow หรือ getAppSession
2. sanitize input ก่อนบันทึก DB
3. ใช้ rate limit กับ endpoint ที่ spam ได้

จุดที่ควรระวัง:
- อย่าดึงข้อมูลจาก client มาตัดสินสิทธิ์โดยตรงถ้าเลี่ยงได้
- route สำคัญต้องมีทั้ง auth + validation + guard

## แนวทางดีบักแบบ Data Flow

เมื่อมี bug ให้ไล่ตามลำดับนี้เสมอ:
1. หน้าไหนเรียก endpoint อะไร
2. endpoint ไหน query table ไหน
3. table/column/index มีครบไหม
4. เงื่อนไขสิทธิ์ทำให้โดน 401/403 หรือไม่
5. ข้อมูลตอบกลับไม่ตรงที่หน้า render คาดหวังหรือไม่

ตัวอย่างรูปแบบไล่:
- UI event -> fetch -> API route -> DB query -> response -> UI state update

ถ้าเขียนบันทึก bug ตาม flow นี้ จะหาสาเหตุได้เร็วขึ้นมาก
